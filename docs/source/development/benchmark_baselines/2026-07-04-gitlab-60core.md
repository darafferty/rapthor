## Interpretation

This is a valid first CI baseline for the compact benchmark scenario. The
runtime resource settings are reaching the parset, and Dask sees the expected
2 worker / 60 thread shape.

The benchmark does not look CPU-allocation limited at the Dask task level. The
main signal is task shape and image-cycle cost:

- The task stream contains only 12 Dask tasks.
- Median Dask task compute time is about 247 seconds, while median Dask report
  duration is about 470 seconds.
- The per-repetition gap between Dask duration and task compute is about
  221 seconds, roughly 47 percent of the Dask report duration.
- Image-sector tasks dominate compute, with source filtering and WSClean the
  largest external command contributors.

The next Dask scalability work should therefore start by explaining and reducing
the idle/orchestration gap, then by splitting or parallelizing image-cycle work
only where the data dependencies and product contracts allow it. Adding more
CPU threads alone is unlikely to improve this benchmark much unless the task
graph exposes more meaningful concurrent work.

## Caveats

- Repetition 1 is about 1-1.5 percent slower than repetitions 2 and 3, so there
  may be a small first-run cache or startup effect.
- Dask task compute time and command profile sums measure different things.
  Dask includes Python work inside tasks and excludes idle/scheduler time;
  command profiles only sum profiled external commands.
- At least one final image cycle logged PyBDSF/no-source and extreme diagnostic
  values. Treat this report as a performance baseline, not as scientific product
  equivalence evidence.
- The downloaded artifact did not include a CI pipeline or job URL. Add it here
  later if the GitLab job page is known.
